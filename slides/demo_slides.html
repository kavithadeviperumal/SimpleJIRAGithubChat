Team Activity Monitor

A natural language chatbot that answers engineering manager questions about what team members are working on — pulling live data from JIRA and GitHub.

Kavithadevi Perumal  ·  2-Day Sprint  ·  June 2026

The Problem

"What is Balu working on?"

Before

  • Open JIRA → search assignee → scan tickets
  • Open GitHub → find profile → scan commits + PRs
  • Manual context-switching for every team member
  • No combined view — mental aggregation by the manager

After — one question

  • Live JIRA tickets with priority + time estimates
  • Recent commits and active pull requests
  • Repos contributed to — no extra API call
  • Single natural language response in seconds
Engineering managers can't answer "what is X working on?" without opening two tabs and doing manual lookups. This chatbot closes that gap.

Design Decisions

3 Key Design Decisions

1 · Intent-based routing

  • Each intent calls only the API it needs
  • Ask about commits → JIRA never called
  • Saves rate limits, cuts latency
  • Adding Slack = one new module + one new intent + one branch

2 · Partial results over hard failures

  • User exists in JIRA but not GitHub?
  • Response shows JIRA tickets — not "user not found"
  • found_set controls which fetchers run
  • Omits sections entirely rather than showing "None"

3 · Validation as single gate + explicit IDs

  • Only validate_member says "user not found" — no API client makes that call
  • Intent-aware: resolves only what the intent needs — get_commits → GitHub login only; get_jira → JIRA accountId only; get_activity → both
  • Failure logic matches intent: get_activity fails only if found in neither system; single-system intents fail if not found in that specific system
  • Returns ResolvedIdentity — passed to every downstream call; nothing downstream ever re-resolves identity
Intent dispatch is an explicit if/elif/elif/elif/else chain. The else is a safety net — future intents missed in the router hit a warning log + friendly message, not a silent crash.

Technical Challenges

Three Technical Challenges

1 · Identity Resolution Across Disparate Systems

The problem

  • JIRA needs accountId — not a display name
  • GitHub needs a login handle — not a profile name
  • Users query in natural language — partial names, aliases
  • git user.name ≠ GitHub profile name → false matches

Resolution strategy

  • Pass 1: fullname:"Kavitha Perumal" user:org
  • Pass 2 fallback: Kavitha Perumal in:login → CamelCase handles
  • JIRA: /user/search?query=name → accountId
  • >1 match + no exact → AmbiguousNameError surfaced to user
2 · Race Condition in Concurrent Resolution

The problem & fix

  • For get_activity, commits + PRs workers run concurrently — both would race to resolve the same user simultaneously
  • Fix: validate_member resolves both IDs before the thread pool starts — workers receive ResolvedIdentity directly, race eliminated
3 · LLM Output as Untrusted Value

The problem & fix

  • LLM extracts member_name from free-form query — but can hallucinate or invent a non-name value
  • Fix: allowlist regex ^[A-Za-z\s'\-\.]{1,100}$ validates it looks like a human name before any downstream use
  • User injection already contained by structured JSON output — LLM returns only intent + member_name
